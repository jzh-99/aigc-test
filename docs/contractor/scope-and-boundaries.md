# 协作范围与边界

这份文档用于帮助大家减少沟通成本、避免返工，并让你可以更聚焦地推进服务鲁棒性和分布式部署工作。这里的边界不是为了限制协作，而是为了让不同方向的负责人都能保持节奏。

## 1. 欢迎重点优化的方向

以下方向非常欢迎你直接调研、提出方案或提交 PR。

### 部署和进程管理

- `setup.sh`
- `deploy.sh`
- `deploy-local.sh`
- `ecosystem.config.cjs`
- `start-api.sh`
- `start-worker.sh`
- `start-web.sh`
- PM2 配置、日志路径、重启策略、日志轮转

### 本地和测试环境

- `docker-compose.yml`
- 本地启动文档
- 本地 mock 环境
- 测试数据初始化
- 开发环境一致性

### 健康检查和可观测性

- `apps/api/src/routes/healthz.ts`
- metrics 接入
- tracing 接入
- 日志规范
- 告警规则
- dashboard
- 服务状态页面

### CI/CD

- GitHub Actions 或其他 CI 流水线
- install / build / lint / test
- migration 检查
- 部署前检查
- 部署后 smoke test
- 回滚流程

### 队列和 Worker 运维

- BullMQ 队列监控
- Worker 并发配置
- retry / backoff 策略建议
- 队列积压告警
- Worker 崩溃恢复
- dead letter queue 方案
- 超时任务监控

### 数据库和缓存运维

- Postgres 备份和恢复
- 连接池 / PgBouncer
- 慢查询和索引建议
- Redis HA 方案
- Redis 内存和连接数监控
- 数据归档建议

## 2. 变更前建议先沟通的方向

如果改动会影响线上业务接口、用户流程、任务状态语义、支付结果或生成任务结果，建议先同步目标和影响范围，再决定怎么改。这样可以减少返工，也方便内部团队一起评估风险。

常见需要先沟通的情况：

- 改动 API 请求/响应格式。
- 改动数据库核心表结构或 migration 策略。
- 改动任务状态流转、重试、失败处理语义。
- 改动支付、额度、订单、账务相关逻辑。
- 改动生成任务的输入输出表现。
- 改动前端核心用户流程。

如果只是部署、监控、日志、CI/CD、健康检查、备份恢复、队列容量规划等基础设施改进，可以直接提出方案或 PR。

## 3. 推荐协作方式

### 提 PR 时建议说明

- 这次变更解决什么稳定性或部署问题。
- 影响哪些服务。
- 是否涉及数据库 migration。
- 是否影响启动命令、环境变量或部署流程。
- 如何验证。
- 如果部署失败，如何回滚。

### 涉及边界模块时建议先发方案

如果变更会影响业务接口、任务语义、用户流程或外部依赖调用方式，可以先发一个简短方案：

```text
目标：
影响范围：
建议改法：
风险：
验证方式：
需要内部团队确认的问题：
```

这样可以减少返工，也方便内部团队快速给出判断。

## 4. 日志和数据处理原则

为了保护用户隐私和系统安全，日志、监控和调试信息默认遵守以下原则：

- 不记录真实密钥。
- 不记录用户敏感内容。
- 不记录可长期访问私有资源的 URL。
- 对上传文件、生成内容、支付信息只记录必要的 ID、状态和耗时。
- 如果为了排障需要更多信息，建议先设计脱敏字段，再和内部团队确认。

## 5. 第一周建议目标

建议第一周先完成这些事情：

1. 本地跑通 API、Worker、Web、Postgres、Redis、MinIO。
2. 记录本地启动过程中遇到的问题，并补充文档。
3. 核对 PM2 启动脚本和服务职责是否一致。
4. 给出部署鲁棒性风险清单。
5. 输出测试服/正式服部署脚本初版。
6. 输出服务器连接文档和部署 runbook 初版。
7. 提供健康检查脚本初版。
8. 提出 CI/CD 初版方案。
9. 提出监控告警初版方案。
10. 选择一个小而明确的稳定性改进提交 PR。

这些任务能帮助你快速熟悉系统，也能让团队尽早看到基础设施方向的改进效果。
